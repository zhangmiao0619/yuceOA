## Bug清单及修复情况

### Bug #1: 金额精度丢失
- **严重程度**: 中
- **描述**: 多条明细金额累加时可能出现浮点数精度问题（如100条0.01元显示为1.0000000000000007）
- **影响**: 影响金额统计准确性
- **修复方案**: 
  - 后端计算总金额时使用 `Math.round(total * 100) / 100` 确保两位小数精度
  - 文件: `backend/src/routes/finance.ts`
- **修复状态**: ✅ 已修复
- **验证结果**: 100条0.01元累加显示为 ¥1.00，精度正确

### Bug #2: 允许极大金额
- **严重程度**: 中
- **描述**: 系统允许创建单条999,999,999.99元的报销明细，缺乏金额上限控制
- **影响**: 可能导致数据异常和潜在风险
- **修复方案**: 
  - 后端添加校验：单条明细金额不能超过100万元
  - 总金额不能超过1000万元
  - 前端InputNumber添加max=1000000限制
  - 文件: `backend/src/routes/finance.ts`, `frontend/src/pages/Finance.tsx`
- **修复状态**: ✅ 已修复
- **验证结果**: 超过100万元的明细被拒绝，提示"单条明细金额不能超过100万元"

### Bug #3: 允许未来日期
- **严重程度**: 低
- **描述**: 系统允许报销日期为未来日期（如2026-12-31），不符合业务逻辑
- **影响**: 不符合实际业务场景
- **修复方案**: 
  - 后端添加校验：报销日期不能为未来日期
  - 前端DatePicker添加disabledDate限制不能选择未来日期
  - 文件: `backend/src/routes/finance.ts`, `frontend/src/pages/Finance.tsx`
- **修复状态**: ✅ 已修复
- **验证结果**: 选择未来日期时提示"报销日期不能为未来日期"

### Bug #4: 允许三位小数
- **严重程度**: 中
- **描述**: 系统允许金额有三位或更多小数位（如100.999），不符合货币精度规范
- **影响**: 可能导致财务报表精度问题
- **修复方案**: 
  - 后端添加校验：检查金额小数位数，超过两位则拒绝
  - 前端InputNumber已设置precision={2}
  - 文件: `backend/src/routes/finance.ts`
- **修复状态**: ✅ 已修复
- **验证结果**: 输入三位小数时提示"金额最多只能有两位小数"
